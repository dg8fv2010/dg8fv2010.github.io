OpenGL-002-OpenGL程序在A卡上运行的问题

By Xue Wentao

发表于 2015-12-11

在A卡上进行GLSL的开发，需要执行更加严格的编程规范。
有一个工程，之前都是在N卡上开发运行的，没有任何问题，但是在A卡上却无法运行。

1.

如果shader文件以UTF-8带签名的形式进行保存，会引入一个标记位。A卡无法识别，导致程序shader编译失败。

2.

有的OpenGL函数在A卡上执行崩溃，这个问题很好定位，也非常好解决。

3.

但是，在程序运行起来之后，却发现绘制的结果是不对的。因为工程比较大，代码结构也比较复杂，只能使用glGetError逐步进行定位。发现程序有的地方会出现error 1282 : invalid operation。但是此处出现的错误是由更之前的引起的。
最后发现，我在fragment shader中使用了sampler2DRect这样一个变量，如果在glUseProgram之前绑定这个变量对应的资源，程序就可以正常运行。但是如果把这个变量类型改成sampler2D，那么在glUseProgram之前之后绑定资源都不会有问题了。  
我觉得，A卡在glUseProgram之前，如果对shader中使用的参数手动绑定了资源，就会使用手动绑定的资源，否则可能会默认绑定一个资源，如果是贴图，就是sampler2D，但如果对应贴图的变量类型是sampler2DRect，绑定就会失败，但是程序不会崩溃，只能用glGetError获取错误信息。
由此可见，在A卡上，在glUseProgram之前需要绑定好所有的资源，但是在N卡上，就不会存在这样的问题。

OpenGL-001-OBJ模型的预处理

By Xue Wentao

发表于 2015-11-28

我在某个项目需要绘制大规模的城市场景，提供的是max文件。但是项目中没有专业的美工人员可以对模型进行优化，原始的模型文件有以下特点。

1. 模型包含很多名称不同但内容相同的材质
2. 模型没有经过合并材质的处理，包含大量的独立小贴图

如果直接拿来使用将会极大地影响绘制效率，因此进行优化。在这里，使用部分场景进行演示。

1. 导出OBJ模型

首先，将max文件转化成OBJ文件，这里先用3dsmax转成OBJ文件，再使用Deep Exploration再转一遍OBJ。因为OBJ格式标准比较多，因此需要通过Deep Explanation转化为统一的格式。我们可以看到，导出的文件包括一个OBJ文件，一个MTL文件，以及非常多的贴图文件。整个场景有1852个Draw Call，50万个面片，以及100多万的顶点。

2. 去除重复材质

在mtl文件中，包含很多内容相同，名称不同的材质。因此需要去除mtl文件中的冗余材质，并修改在obj文件中的引用。

3. 拆分材质

打开OBJ文件可以看到，在一个Group内，包括v（顶点坐标），vt（纹理坐标），vn（法线），usemtl（材质），f（面片）。一个OBJ文件中，一个的Group可能包含许多的材质。需要将材质进行拆分，使得每一个Group只包含一个材质。

4. 合并贴图

建模人员使用了大量的独立的小贴图，增加了绘制批次。因此，需要将这些小贴图合并成大贴图。其实就是一个纹理坐标映射的过程，首先定义目标贴图，把小贴图复制到目标贴图上，再重新计算对应的纹理坐标，直至目标贴图被填满。保存目标贴图，并重复上面的过程。

5. 重新拆分材质

将贴图进行合并之后，材质又会被重新打乱，因此需要再次对材质进行合并，重复第3步。

模型经过处理，可以发现Draw Call的数量下降到了952，效果显著。

模板特化是怎样工作的？
当我们在实现一个类或函数的模板时，当被一个新的数据类型或新的数据类型集合（以防多个模板参数）使用时，编译器都会创建一份类或函数的拷贝。
当一个特化版本出现时，编译器首先检查特化版本，再检查主模板。
Compiler first checks with the most specialized version by matching the passed
parameter with the data type(s) specified in a specialized version.
但出现一个特化版本时，编译器首先检查特化版本再检查主模板。编译器会首先会对传入参数的类型进行匹配。
